[Protocol Buffers][Protocol_Buffers]，又名 `protobuf`，是谷歌内部使用的一种**数据交换格式**，于 2008 年开源 (proto2), 2016 年发布 `proto3`。后被业界广泛采用发展至今。

![Protobuf timeline][protobuf_timeline]

<!-- more -->

### 简介

谷歌在[官方文档][Protocol_Buffers]上说 Protocol buffers 是一种与语言无关、与平台无关的，用于序列化结构化数据的可扩展机制。与之相比，我更喜欢其在 [Github 上的描述][protobuf] `Protocol Buffers - Google's data interchange format` —— 一种数据交换格式，清楚明了。用于序列化结构化数据的可扩展机制还自罢了，说的过去但太细节。而语言无关、平台无关更是无用的废话，媲美门是可以开的，饭是可以吃的，空气是可以呼吸的，……。正确而无用。试问那个数据交换格式不是这样？

> Protocol buffers are a language-neutral, platform-neutral extensible mechanism for serializing structured data.

#### 数据交换格式

为了进程、程序、客户服务器之间通信，人们定义了一批数据交换格式以应对不同的场景。和 Protocol Buffers 同级，近期又比较流行的有 [`JSON`][JSON]，[`XML`][XML] 和 [`YAML`][YAML]，相信大家都不陌生。Protocol Buffers 类似，只是定义的格式和编码方式不同而已。和 `JSON` 等相比，Protocol Buffers：

- 以二进制存储，体积小
- 编解码速度快
- 可读性差

#### 序列化与反序列化

所谓[序列化与反序列化][Serialization]，是指将数据结构或对象状态转换为可以存储的格式(例如，存储在文件或内存缓冲区中，或通过网络连接链接传输)，并稍后在相同或另一个计算机环境中重建格式的过程。

一般来说，先由协议方定义数据交换格式，再由编程语言/开源社区予以支持。某些流行格式的序列化与反序列化甚至被包含在语言模块中，例如 [Ruby 的 JSON 模块][Ruby_JSON]。公共的解析模块只能把 JSON 反序列化成内置的格式，再由程序员映射到具体模型上。以 Ruby 为例，先将序列化的数据解析成内置结构 `Hash`，然后再由程序员转存到自己的模型上。

```ruby
require 'json'
json = '{"name":"zddhub","age":18,"blog":"https://zddhub.com","income":18.125}' # Serialized JSON
ruby_hash = JSON.parse(json)
ruby_hash # => {"name"=>"zddhub", "age"=>18, "blog"=>"https://zddhub.com", "income"=>18.125}
ruby.class # => Hash
author = Author.new(ruby_hash["name"], ruby_hash["age"], ruby_hash["blog"], ruby_bash["income"]) # Mapping
author.class # => Author
```

Protocol Buffers 做的更直接。它允许程序员直接使用 Protocol Buffers 定义的接口描述语言来定义模型，之后通过 Protocol Compiler `protoc` 来生成对应语言的模型代码，(如上例中的 Author)。该模型负责序列化与反序列化，而且不需要再映射或者转存，可直接使用。

`protoc` 起初只支持谷歌自家内部使用的六种语言，后发展到 [10 种][Protobuf_Runtime]，其它语言由[第三方提供][Protobuf_Third_Party]。

通常，我们通过编码来实现序列化，通过解码来反序列化。在本文的语境中，序列化反序列化与编码解码是一个意思。

综上，Protocol Buffers 做为一种数据交换格式，主要负责两件事：

- 定义了一种接口描述语言 (Interface Description Language)，用来描述需要交换的数据结构
- 定义了一种编码解码方式，用来序列化反序列化 Protobuf 数据

### Protocol Buffers 接口描述语言

Protocol Buffers 接口描述语言描述用于交换的数据结构，语法和现今大部分编程语言的数据结构一致。一般来说，编程语言里的数据结构主要有：

- 基础类型：Integer，Float，Double，Boolean，Null, Nil, ……
- 字符串
- 集合类：Array，Map, List, Hash, Dictionary，……
- 自定义结构：Class，Enum，Struct，Type，Union, ……

protobuf 也类似，只不过关键词不同，变不到哪里去。具体的语法可以参考 [Language Guide (proto3)][Proto3_Guide]。若没有特别说明，本文内容以 proto3 为准。

我一般会用一个 CheatSheet 来快速了解一门新语言的写法，当然 protobuf 也有，这里是 Github Gist 上 proto3 的一个 [CheatSheet][Proto3_CheatSheet]，大家可以参考。

CheatSheet 也有两百多行，看着很累，也远远超出了我的脑容量。我一般只能记住一成，所以总结了一个简化版，如下：

```protobuf
/* proto3_syntax.proto  */
syntax = "proto3"; // 1. 指定使用 proto3，如不指定则默认为 proto2

message Message { // 2. 定义数据结构 Message，用关键字 message 表示
  int32 id = 1; // 3. 定义 Message.id，类型为 int32. 其它 Integer 的定义类似。并定义了字段号，字段号用于编码，最小字段号为1
  string str = 3; // 4. string
  repeated int64 arrays = 4; // 5. arrays，长度 >= 0

  enum Answer { // 6. 定义 enum 类
    YES = 0; // 7. enum 第一个元素的字段号必须为 0
    NO = 1;
  }
  Answer answer = 16;

  message InnerMessage {  // 8. message 可以嵌套
    float test_float = 1; // 9. 定义 test_float. 内部 message 属于新的 message。字段号可重新开始
    map<string, string> map = 1; // 10. 定义 map
  }
  InnerMessage inner_message = 300;

  oneof test_oneof { // 11. 类似 C 语言 的 union，is_true 和 bytes 共享内存，同时只能设置其中之一
    bool is_true = 5; // 12. 只是 Message 的特殊属性，字段号和 Message 共享，不可冲突
    bytes bytes = 6;  // 13. bytes，类似 string
  }

  int32 max_field_number = 536870911; // 14. 该属性使用最大的字段号
}
```

proto3 的所有属性都是可选的。repeated 属性可包含0个或多个值，类似 Array。和其它语言最明显的不同是，Protocol Buffers 的每个属性包含一个字段号。该字段号在同一个 message 内不可重复。最小的字段号是 1，最大是 2^29-1（536,870,911）。

在此强烈建议大家通读 [Language Guide (proto3)][Proto3_Guide]，不要像我一样投机取巧，此乃做学问之大忌。

你真优秀能坚持读到这里！接下来让我们看一下本文的核心部分，Protocol Buffers 的编码和解码。

### Protocol Buffers 编码

先直观的感受一下流行格式的编码结果。假如有如下 `Author` 数据，我们如何来编码呢？

```rb

```

```json

```

那么 protobuf 到底是如何来编码的呢？

#### Base 128 Varints

Protobuf 使用了一种叫做 `Base 128 Varints` 的编码方法。翻译一下就是 **128 进制变长整型编码**方法。该方法使用一个到多个字节序列化整型数据，整型数值越小，所占用的字节越少。逢 128 进 1。小于 128 时只用一个字节表示，128 用两个字节表示，128x128 用三个字节表示，……。每个字节的最高位 (most significant bit - msb) 用来表示该字节是不是最后一个字节。最高位为 0 时表示当前字节是最后一个字节，为 1 时表示还存在后续字节，最后一个字节放在最后。所以说 `Base 128 Varints` 是一种自带分隔符的编码方式。

![Base 128 Varints][base128_varints]

举个例子，上述年龄 18，小于 128，只要一个字节，最高位为 0。编码结果为:

```
00010010
```

再比如 320，需要两个字节：

```
11000000 00000010
```

![Varints 320][varints_320]

#### Protocol Buffers 编码

上面例子中说到 Author 的 protobuf 编码中所有 key 值，比如 name, age, blog 和 income 都不可见，是因为 protobuf 根本没有把这些值编码。而使用了自定义的字段号。编码解码时都需要参考 .proto 文件。

这是 Author 的定义：

```protobuf
message Author {
  string name = 1;
  int32 age = 2;
  string blog = 3;
  double income = 4;
}
```

实际上 Protocol Buffers 省去了消息名和属性名，只编码了以下内容，暂时使用三元组表示(Field Number, Type, Value)：

```
1 string zddhub
2 int32 18
3 string https://zddhub.com
4 double 18.125
```

如果把 字段号+类型 看做 key 的话，那么它也是一种 key-value 结构。通常说的 key 是指 name, age, blog 或者 income，为了区分，我们把 字段号+类型 叫做 tag。

Protobuf 对 Type 也做了分类和编码，用 3 bit 表示，进一步压缩了编码空间。如下表所示：

| Type | Meaning | Used For |
|------|---------|----------|
| 0 | Varint | int32, int64, uint32, uint64, sint32, sint64, bool, enum |
| 1 | 64-bit | fixed64, sfixed64, double |
| 2 | Length-delimited | string, bytes, embedded messages, packed repeated fields |
| 3 | Start group | groups (deprecated) |
| 4 | End group | groups (deprecated) |
| 5 | 32-bit | fixed32, sfixed32, float |
| 6 | Reserved ||
| 7 | Reserved ||

Protobuf 的逻辑结构如下：

![Protobuf Tag Value][tag_value]

如果用一个字节表示 Tag 的话，实际上只有 4 bit 存储字段号，所以一个字节可以表示的字段号从 1 到 2^4 -1 = 15，字段号不包括 0 。从 16 到 2047 (2^11 -1) 需要两个字节。最大字段号 536,870,911 (2^29-1) 需要 5 个字节 (最高四位不用)。使用较小的字段号能节省存储空间。

![Field Number][field_number]

如果知道了字段号，对应到 proto 文件，能很容易的知道字段类型和字段名，为什么要把 Type 编码呢？其实这里的 Type 并不能区分当前字段是 int32 还是 int64。Type 更重要的是标记了内容该以何种方式编码。

根据 Type，去掉被 proto3 废弃的两个类型。我们可以把 Protobuf 的编码方式分为三类：

- Tag + Varints (Type = 0)
- Tag + Value (Type = 1 或者 Type = 5)
- Tag + Length + Value (Type = 2)

##### Tag + Varints (Type = 0)

让我们从年龄开始编码，年龄采用 `Tag + Varints` 的编码方式。根据 Author 的定义，年龄的字段号是 2，类型是 0，年龄 18，把 Tag 和 年龄都按 Varints 编码后既得编码结果:

```
00010000 00010010
```

![Age Encoding][age_18]

##### Tag + Value (Type = 1 或者 Type = 5)

收入采用 `Tag + Value` 的编码方式。根据 Author 的定义，收入的字段号是 4，类型是 1。Value 直接采用 double 双精度浮点数的编码方式 [IEEE 754][IEEE_754]。收入值 18.125 编码后的结果为：

```
2100 0000 0000 2032 40
```

共 9 个字节，其中 Tag 一个字节, tag 编码为 `00100001` (0x21)，double 8 个字节。

这里有必要聊一下浮点数的编码方法：

根据 [IEEE 754][IEEE_754] 标准，浮点数在计算机中用三部分来表示，符号位 (Sign) + 指数部分 (Exponent) + 尾数部分 (Mantissa)。其中的指数部分（E）采用偏置码（biased）的形式来表示正负指数，若 E<127 (单精度) 或者 E<1023 (双精度) 则为负的指数，否则为非负的指数。尾数部分 (M) 存储的是当把一个浮点数规范化表示后的1.zozooz...（二进制的）形式的zozooz的部分的比特串。

其中，单精度浮点数 float 采用 4 个字节 32 位来表示，符号位 1bit，指数部分 8bit，尾数部分 23bit。双精度浮点数 double 采用 8 个字节 64 位来表示，符号位 1bit，指数部分 11bit，尾数部分 52bit。

![Floating-Point Arithmetic][Floating_Point_Arithmetic]

现在我们来求双精度的 18.125 在计算机中的表示：

1. 先把整数部分用二进制表示，18的二进制是 10010
2. 再把小树部分用二进制表示，0.125 的二进制是 001

   ```txt
   0.125 * 2 = 0.25, 取 0
   0.25 * 2 = 0.5    取 0
   0.5 * 2 = 1       取 1
   结束
   ```
3. 转成科学计数法，并规范化 (10010.001) * 2^0 = 1.0010001 * 2^4
4. 确定 S，E 和 M 的值。正整数 S = 0，指数为 4，基数为 1023，所以指数部分 E = 1023 + 4 = 1027, 二进制 1000 0000 000011, 尾数 M = 0010001
5. 按顺序写出 18.125 的二进制表示，尾数末尾用 0 补齐。`01000000 00110010 00100000 00000000 00000000 00000000 00000000 00000000`。
   十六进制为 `4032 2000 0000 0000`。

这里有个在线的[二进制转换工具][Binary_Converter]，大家无聊了可以玩玩。

Protobuf 存储浮点数是采用小端序(Little Endian)，低地址端存放低位字节，所以其存储为 `0000 0000 0020 3240`。加上 tag `0x21`。终得 income 的编码结果：

```
2100 0000 0000 2032 40
```

##### Tag + Length + Value (Type = 2)

名字采用 `Tag + Length + Value` 的编码方式。根据 Author 的定义，名字的字段号是 1，类型是 2，Length 为字符串的长度 6，Value 为字符串的 ASCII 码。编码后的结果为：

```0a06 7a64 6468 7562```

![Name Encoding][name_zddhub]

值得注意的是，Length 也用 Varints 来编码。同样的做法，我们可以编码 blog，编码结果为：

```
1a12 6874 7470 733a 2f2f 7a64 6468 7562 2e63 6f6d
```

至此，我们了解了 Protobuf 使用的三种编码方式。编码了 Author 的四个字段，连起来即为开头给出的 Author 的编码结果。

##### 编码顺序和默认值

这里的编码顺序是很有意思的。每个 Tag-Value 对有严格的顺序要求，但 Author 的四个字段之间可以随意组合 4！种方式。而且中间可以有任意多个未知字段，只要未知字段按照 Protobuf 的字段编码就好。这也是 Protobuf 兼容新老版本的奥秘。

![Author Combination][Author_Combination]

如上图所示，使用了和文章开头不同的编码顺序，但都是正确的编码。第二排多了一个 Field Name 是 5 ，Type 也是 5 的字段，有可能是 float，fixed32 或者 sfixed32。但当前版本 Author 的 Protobuf 消息里没有对该字段的定义，所以具体是那一个谁也不知道，被当作未知字段扔掉。

值得注意的是第二排的编码中缺少了 blog。proto3 中的每个字段都是可选的。所以它仍然可以被 `protoc` 生成的代码来解析，并且对于缺省的字段，会自动使用默认值。下面是不同字段的默认值：

Fields   | Default Value
---------| -------------
strings  | empty string
bytes    | empty bytes
bools    | false
number   | zero
enum     | first enum value, field number must be 0
message  | not set, Its exact value is language-dependent
repeated | empty list

### Protocol Buffers 解码

知道了编码规则，解码就容易了，这里尝试解码一个例子。已知经 Protobuf 编码后的二进制如下：

```
00000000: 00001000 00011011 00011010 00001011 01001000 01100101  ....He
00000006: 01101100 01101100 01101111 00100000 01010111 01101111  llo Wo
0000000c: 01110010 01101100 01100100 00100010 00000010 00000001  rld"..
00000012: 00000010 00101000 00000000 10000000 00000001 00000001  .(....
00000018: 11100010 00010010 00000101 00001101 00001010 11010111  ......
0000001e: 00110011 01000001 11111000 11111111 11111111 11111111  3A....
00000024: 00001111 00010010
```

首先，Protobuf 的编码是按照 Tag + Value 和 Tag + Length + Value 组合的，那么第一个字节永远是 Tag。

- 第 1 个字节 `00001000`, 最高位是 0，说明该 Tag 仅用一个字节表示。字段码是 1，类型是 Varint，说明该字段值是这里面的一种 int32, int64, uint32, uint64, sint32, sint64, bool, enum。虽然我们不知道是哪一种，但是我们知道，该字段值只用 Varint 编码。那么我们继续读第二个字节。
- 第 2 个字节 `00011011`, 最高位是 0，Varint 的编码结束。那么我们得到第一个 Tag-Value 对，暂时使用一个三元组来表示 (1，Varint，0x1B)。这里使用 16 进制来表示内容，而没有使用 27。因为没有任何信息能说明它是整型还是 enum。
- 第 3 个字节 `00011010`, 上一个 Tag + Value 结束，这还是一个一字节的 Tag。字段码是 3，类型是 Length-delimited。该类型使用 `Tag + Length + Value (Type = 2)` 的方式编码。那么接下来的字节表示长度。
- 第 4 个字节 `00001011`, 根据第 3 个字节推断，从这个字节开始表示长度。高位为 0，说明该字节仅用一个字节表示长度，长度为 11。说明后面用 11 个字节表示该字段的值。顺势读取 11 个字节。
- 第 5 ～ 15 个字节 `01001000 01100101 01101100 01101100 01101111 00100000 01010111 01101111 01110010 01101100 01100100`, 为字段码为 3 的字段的内容。这 11 个字符正好是 `Hello World` 的 ASCII 码，我们可以大胆猜测该字段是 string (3, Length-delimited, Hello World)，但是没有任何证据，所以暂时表示为 (3, Length-delimited, 0x4865 6c6c 6f20 576f 726c 64)。
- 第 16 个字节 `00100010`，字段码是 4，类型是 Length-delimited。那么
- 第 17 个字节 `00000010`, 为长度，长度为 2。
- 第 18 ~ 19 个字节 `00000001 00000010` 为该字段的内容，(4, Length-delimited, 0x01 02)。
- 第 20 个字节 `00101000`, 字节码是 5，类型是 Varint。
- 第 21 个字节 `00000000`, 该字段值为 0。(5, Varint, 0x0)，下一个字节为 Tag。
- 第 22 个字节 `10000000`, 该字节表示 Tag，但是最高位为 1，说明 Tag 超过一个字节，继续读下一个
- 第 23 个字节 `00000001`, 此时最高位为 0，Tag 结束，该 Tag 用两个字节表示 `00000001 10000000`，注意这里的顺序。去掉两个字节最高位 `0000001 0000000` 得实际的 Tag 值, 字段码为 16 (`00000010000`), 该字段的类型为 Varint。
- 第 24 个字节 `00000001`, 为该字段的值。(16, Varint 0x1), 下一个 Tag。
- 第 25 个字节 `11100010`, 该字节表示 Tag，但是最高位为也为 1，继续读下一个
- 第 26 个字节 `00010010`, 此时最高位为 0，Tag 结束。该 Tag 用两个字节表示 `00010010 11100010`，去掉两个字节最高位 `0010010 1100010` 得实际的 Tag 值，字段码为 300 (`00100101100`)，该字段的类型是 Length-delimited。
- 第 27 个字节 `00000101`, 为长度。长度为 5。
- 第 28 ～ 32 个字节 `00001101 00001010 11010111 00110011 01000001`, 读取 5 个字节解码出该字段为 (300, Length-delimited, 0x0d0a d733 41)
- 第 33 个字节 `11111000` 为 Tag 的开始，
- 第 34 ～ 37 个字节 `11111111 11111111 11111111 00001111` 一直读到高位为 0 结束，该 Tag 使用 5 个字节表示，逆序写出 `00001111 11111111 11111111 11111111 11111000`，去掉最高位的 0 和 末尾的 Type，字段码全为 1，2^29 = 536,870,911。是被允许的最大字段号。类型是 Varint。
- 第 38 个字节 `00010010`, 高位为 0，该字节即为最后一个字段的值。(536,870,911, Varint, 0x12)

至此，我们解码完了所有内容。让我们看看解码后的内容：

```
(1，Varint，0x1B)
(3, Length-delimited, 0x4865 6c6c 6f20 576f 726c 64) # Hello World
(4, Length-delimited, 0x01 02)
(5, Varint, 0x0)
(16, Varint 0x1)
(300, Length-delimited, 0x0d0a d733 41)
(536,870,911, Varint, 0x12)
```

我们解码出了 7 个 Tag/Value 对，知道每个字段码和值。由于不知道其类型，所以值也并没有被完全解码。这个时候就需要参照 .proto 的定义了。我们参考 proto 的定义文件，程序解码时使用 .desc 文件，(.desc 是使用 `protoc`生成的 protobuf 描述文件，二进制存储，方便程序解码)。

我想你也猜到了，我们解码内容正是本文开头的 Message。

```protobuf
message Message {
  int32 id = 1;
  string str = 3;
  repeated int64 arrays = 4;
  enum Answer {
    YES = 0;
    NO = 1;
  }
  Answer answer = 16;
  message InnerMessage {
    float test_float = 1;
  }
  InnerMessage inner_message = 300;
  oneof test_oneof {
    bool is_true = 5;
    bytes bytes = 6;
  }
  int32 max_field_number = 536870911; // 14. 该属性使用最大的字段号
}
```

参考该定义文件，我们进一步解码:

```java
Message {
  int32 id = 27 // (1，Varint，0x1B)
  string str = "Hello World" // (3, Length-delimited, 0x4865 6c6c 6f20 576f 726c 64)
  int64 arrays = [1, 2] // (4, Length-delimited, 0x01 02)
  bool is_true = false // (5, Varint, 0x0)
  Answer answer = Answer.No // (16, Varint 0x1)
  InnerMessage inner_message = 0x0d0a d733 41 // (300, Length-delimited, 0x0d0a d733 41)
  int32 max_field_number = 18 // (536,870,911, Varint, 0x12)
}
```

我们解码了大部分数据，除了 InnerMessage。其内容为 `0x0d0a d733 41`，二进制如下：

```
00001101 00001010 11010111 00110011 01000001
```

- 第 1 个字节 `00001101` 仍然表示 Tag，其中字段号为 1，类型为 5，采用 Tag + Value (Type = 5) 方式编码。其内容用四个字节表示。读取四个字节。
- 第 2 ～ 5 个字节 `00001010 11010111 00110011 01000001` 表示该字段的值。该字段为 (1, 32-bit, 0x0ad7 3341)

查定义得该字段为 float，单精度浮点类型，只要翻译出其值即可。根据解码部分可知: Protobuf 存储浮点数是采用小端序(Little Endian)，低地址端存放低位字节，所以存储值为 `00001010 11010111 00110011 01000001` 的浮点数实际值为:

```
01000001 00110011 11010111 00001010
```

单精度浮点类型首位为符号位 S = 0，紧接着 8bit 为指数部分 E = 130 (10000010)，剩余 23bit 为尾数 (M = `0110011 11010111 00001010`)。那么该浮点数的值为：

```
Value = (-1)^0 * (1.01100111101011100001010) * 2^(130-127) = 1011.00111101011100001010
```

其中整数部分为 1011 = 11，小数部分为：00111101011100001010。那小数部分的二进制如何转十进制呢？编码部分把 0.125 转成二进制时每次都乘了，我们这次做除法就好了。小数部分依次除以2^1, 2^2, 2^3, 2^4, ..., 即可。

那么小数部分 = 1/2^3 + 1/2^4 + 1/2^5 + 1/2^6 +  1/2^8 + 1/2^10 + 1/2^11 + 1/2^12 + 1/2^17 + 1/2^19 = 0.2399997711181640625 ～= 0.24

得该浮点数的值为 11.24。

至此，我们解析出了所有字段：

```java
Message {
  int32 id = 27 // (1，Varint，0x1B)
  string str = "Hello World" // (3, Length-delimited, 0x4865 6c6c 6f20 576f 726c 64)
  int64 arrays = [1, 2] // (4, Length-delimited, 0x01 02)
  bool is_true = false // (5, Varint, 0x0)
  Answer answer = Answer.No // (16, Varint 0x1)
  InnerMessage inner_message { // (300, Length-delimited, 0x0d0a d733 41)
    float test_float = 11.24 // (1, 32-bit, 0x0ad7 3341)
  }
  int32 max_field_number = 18 // (536,870,911, Varint, 0x12)
}
```

从这里也可以看出，想要正确的理解数据，需要手握 .proto 。这种省略 key 值 (如果不省略会不会被人诟病是 [BJSON][BJSON] 呢) 加上紧凑的编码方式，压缩了存储空间。同时也降低了可读性，有一定的加密作用。没有 protobuf 定义的数据，解码了也存在理解困难的问题。而同 JSON 等文本型数据交换格式相比，解码速度提高显著。protobuf 的解码速度可达 4.62609GB/s，参见 [Protobuf performance][Protobuf_performance]。实际的解码并不像人工这样，在生成的解码文件里，预先根据 tag 定好了类型，例如 java 版本的一个解析文件代码片段如下，大家可以感受一下：

message Award {
